1
Persistencia en IPFS (Content Addressing)

Carga de activos en la red IPFS. El identificador de contenido (CID) garantiza la integridad del archivo mediante hashing criptográfico.

2
Estructuración de Metadatos (JSON)

3
Acuñación (Minting) On-Chain

Ejecución de la función de acuñación en el Smart Contract, vinculando el ID del token con el URI del metadato en IPFS.

Gestión de Nodos con Pinata

Pinata actúa como un servicio de "pinning" profesional, garantizando que los activos críticos permanezcan accesibles en la red IPFS de forma persistente.

Sin un servicio de pinning, los archivos en IPFS pueden ser eliminados por el proceso de "Garbage Collection" si no tienen suficiente tráfico o nodos activos.

Integración vía SDK/API
// Ejemplo de carga programática
const pinata = new PinataSDK({ 
  pinataJWTKey: 'AUTH_TOKEN' 
});

const result = await pinata.pinFileToIPFS(fileBuffer);
console.log("CID Generado:", result.IpfsHash);

Diferenciación de Estándares: ERC-721 vs ERC-1155

ERC-721 (Non-Fungible)

Implementación de activos estrictamente únicos. Cada ID de token mapea a un único propietario y URI.

// Cada acuñación genera un token ID incremental único
function mint(address to) public onlyOwner {
    uint256 tokenId = _tokenIdCounter.current();
    _safeMint(to, tokenId);
    _tokenIdCounter.increment();
}

ERC-1155 (Multi-Token)

Estándar eficiente que permite la gestión de múltiples tipos de tokens (fungibles y no fungibles) en un solo contrato.

// Un solo ID puede representar una colección de N elementos
function mintBatch(address to, uint256 id, uint256 amount) public {
    _mint(to, id, amount, "");
}

Fundamentos de Contenido Direccionable

Arquitectura Centralizada (Web2):

Dependencia de URLs basadas en ubicación (ej: `servidor.com/img.jpg`). Si la ubicación cambia o el servidor falla, el enlace se rompe.

Arquitectura Descentralizada (Web3):

Uso de identificadores basados en el contenido (Hashing). El enlace es el archivo mismo; si el contenido no cambia, el CID permanece inalterable.
