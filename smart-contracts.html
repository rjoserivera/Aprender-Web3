Ejemplo: Almacén de Datos Persistente 
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.18;

/**
 * @title AlmacenProfesional
 * @dev Implementación de almacenamiento de estado con control de acceso.
 */
contract AlmacenProfesional {
    string private _mensaje;
    address public owner;

    event MensajeActualizado(string nuevoMensaje, address autor);

    constructor() {
        owner = msg.sender;
    }

    /**
     * @dev Actualiza el mensaje en el almacenamiento del contrato.
     * Requiere el pago de gas para la modificación del estado.
     */
    function actualizar(string memory nuevo) public {
        _mensaje = nuevo;
        emit MensajeActualizado(nuevo, msg.sender);
    }

    /**
     * @dev Retorna el mensaje almacenado.
     * Función de vista (view) que no consume gas.
     */
    function consultar() public view returns (string memory) {
        return _mensaje;
    }
}

Anatomía de un Protocolo Robusto

1. Gestión de Estado (Mappings)

Los mappings son estructuras de datos altamente eficientes para el almacenamiento de pares clave-valor en la memoria persistente de la blockchain.

// Mapping de Direcciones a Balances
mapping(address => uint256) public saldos;

function depositar() public payable {
    saldos[msg.sender] += msg.value;
}

2. Control de Acceso (Modifiers)

Los modifiers permiten la reutilización de lógica de validación, garantizando la seguridad en la ejecución de funciones sensibles.

modifier onlyOwner() {
    require(msg.sender == owner, "Acceso Denegado");
    _; // Continuación de la ejecución
}

function transferirPropiedad(address newOwner) public onlyOwner {
    owner = newOwner;
}

3. Emisión de Eventos (Logging)

Los eventos permiten que las aplicaciones externas (dApps) reaccionen a cambios de estado en el contrato mediante la escucha de logs indexados.

// En Solidity: Definición de Log
event Transfer(address indexed from, address indexed to, uint256 value);

function _transfer(address to, uint256 amount) internal {
    emit Transfer(msg.sender, to, amount);
}
// En Ethers.js: Escucha de Eventos
contract.on("Transfer", (from, to, value) => {
    console.log(`Transferencia detectada: ${from} -> ${to}`);
    refreshUI();
});

La EVM y la Optimización de Gas

La **Ethereum Virtual Machine (EVM)** es el motor de ejecución global. Cada instrucción de bajo nivel (Opcode) tiene un coste asociado en Gas.

La optimización del almacenamiento (Storage) es crítica, ya que escribir en la blockchain es la operación más costosa. El uso de tipos de datos adecuados (ej: uint128 empaquetado) puede reducir significativamente los costes de transacción.

BEST PRACTICES: GAS OPTIMIZATION
  • • Utilizar external para funciones que solo se invocan fuera del contrato.
  • • Implementar calldata en lugar de memory para parámetros de lectura.
  • • Minimizar el uso de bucles sobre estructuras de datos dinámicas.
